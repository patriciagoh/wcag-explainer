Phase 2 — Enriching the Dataset with Claude

Phase 1 collected the facts. Phase 2 is where Claude turns each of those raw criteria into something an engineer can actually learn from — plain-English summaries, real pass/fail code, and grounded "common mistakes" — without ever inventing the facts underneath.

01 Why this phase exists

Official WCAG text is precise but impenetrable to a working engineer. Phase 2's job is to translate each criterion into actionable guidance — and to do it with an LLM, because that translation is judgement, not data.

  • ◆ Turn legalese into action. "Non-text content has a text alternative…" becomes "every image needs alt text; decorative ones get alt=""." That rewrite is exactly what an LLM is good at.
  • ◆ Teach with contrast. Each criterion gets a passing and a failing code example. Seeing both side by side teaches faster than prose.
  • ◆ Ground every claim. "Common mistakes" are tied to the criterion's real axe-core rules from Phase 1, and the only non-synthesized example reuses a real W3C technique snippet. Facts stay sourced.
  • ◆ Be cacheable & resumable. Each criterion is hashed against its inputs. Re-runs skip anything unchanged, so a refresh only re-enriches what actually drifted — and a session can stop and resume safely.

02 Key decisions we made

Phase 2 runs inside a Claude Code session rather than as an automated script — Claude is the "LLM rewrite" step. A few choices kept that honest, reproducible, and reviewable.

Claude is the generator, in-session

Claude reads each raw criterion plus the prompt templates and writes the fields directly — no external API call in the loop.

Why: the rewrite needs context and judgement; doing it in-session keeps it transparent and steerable, one principle at a time.

A deterministic inputHash per criterion

SHA-256 over the normalized {officialText, scrapedExamples, axeRules} with recursively sorted keys.

Why: a stable hash lets any future run reproduce it, compare, and skip criteria whose facts haven't changed — true caching and resume.

Batch by principle, commit between

Work proceeded in four batches — Perceivable (29), Operable (34), Understandable (21), Robust (3) — committing after each.

Why: keeps each session's context manageable and the work resumable; a stopped session loses nothing already committed.

Real example when we have one, synthesize otherwise

Only 1.1.1 had scraped examples (7, all "sufficient"). Its pass reuses the real W3C H37 snippet; its fail — and every other example — is synthesized.

Why: prefer sourced truth, but never ship a criterion without a teaching pair. Synthesized examples are marked source:"synthesized", honestly.

Pick the most natural language per example

JSX for component/keyboard patterns, CSS for contrast/motion/target-size, HTML for <meta>/<title>.

Why: forcing everything into JSX would misrepresent CSS-level criteria like contrast or reflow.

Validate against the real schema before claiming done

Every entry was parsed through the same Zod criterionSchema the merge uses, plus the full --merge and test suite.

Why: evidence before assertions — a cache file only counts as done if the production validator accepts it.

03 What we built

For every one of the 87 criteria, a single cache file with six keys. The fields below are what Phase 2 produces and Phase 3 merges into the shipped dataset.

grounding key inputHash

SHA-256 of the criterion's facts. Drives skip/resume on re-runs.

prompt → plain-english.md plainEnglish

2–4 plain sentences explaining what the criterion requires and why.

prompt → why-it-matters.md whyItMatters

One concrete sentence on who is affected when it's violated.

prompt → quick-check.md quickCheck

A one-line test an engineer can run now (DevTools / axe).

prompt → common-mistakes.md commonMistakes[]

3–5 concrete failures, grounded in the criterion's axe rules.

prompt → html-to-jsx.md codeExamples[]

One pass + one fail snippet, each labelled and sourced.

The deliverable: 87 files at scripts/cache/{id}.json, committed in four batches, then merged into template/src/data/wcag-criteria.json — 87 criteria, fully Zod-validated, every one with a pass and a fail example.

04 How it runs, per criterion

For each criterion in criteria-raw.json, Claude follows the same loop from enrich-with-claude.md.

  1. Hash & check the cache. inputHash
    Compute the criterion's inputHash. If a cache file exists and the hash matches, skip — already fresh.
  2. Generate the three short fields. plain-english · why-it-matters · quick-check
    Fill each prompt template with the criterion's id, title, level, and official text, then write the summary, the impact sentence, and the quick check.
  3. Ground the common mistakes. common-mistakes.md
    Feed the criterion's axe-rule descriptions into the template so the 3–5 mistakes reference real, flagged patterns — not abstractions.
  4. Build the pass/fail pair. html-to-jsx.md
    Pick the best scraped pass & fail examples; convert HTML → idiomatic JSX. Where none exist, synthesize a realistic pair and mark it "synthesized".
  5. Write & commit the batch. cache/{id}.json → git
    Serialize the entry, and after a whole principle, commit. Then a final --merge validates and ships the dataset.

06 What an enriched record looks like

The actual scripts/cache/1.1.1.json Claude produced. Compare it to the Phase 1 raw record: same facts underneath, now wrapped in teachable guidance. 

// scripts/cache/1.1.1.json
{
  "inputHash": "19b05a8e…945d791",        // gate for skip/resume
  "plainEnglish": "Every image, icon, chart… needs a text alternative…",
  "whyItMatters": "Screen reader users… get nothing useful when an image has no alt text…",
  "quickCheck": "In Chrome DevTools, inspect each <img>… or run the axe DevTools extension.",
  "commonMistakes": [                          // 3–5, grounded in axe rules
    "Leaving the alt attribute off an <img> entirely…",
    "Using a <div onClick> as a button with no aria-label…"

    { "kind": "pass", "language": "jsx",
      "code": "<img src=\"newsletter.gif\" alt=\"Free newsletter…\" />",
      "source": "H37" },                       // real W3C technique
    { "kind": "fail", "language": "jsx",
      "code": "<img src=\"newsletter.gif\" />",
      "source": "synthesized" }
  ]
}

07 How to run Phase 2 yourself

Like Phase 1, this is a skill-author task — only needed when rebuilding or refreshing the dataset. End users of the app never run it.

Claude Code · skill directory
# open a Claude Code session in the skill directory, then say:
Follow scripts/enrich-with-claude.md. Start with principle 1.

Claude hashes each raw criterion, fills the prompt templates,
and writes scripts/cache/{id}.json — committing after each principle.

# when all four principles are done, merge & validate:
cd scripts
npm run build:dataset -- --merge
Wrote template/src/data/wcag-criteria.json with 87 criteria

Resumable: the inputHash gate means a re-run skips any criterion whose facts are unchanged — so a refresh after Phase 1 drift only re-enriches what actually changed.

Run a batch

"Start with principle 1" — then 2, 3, 4. One principle per session keeps context tight.

Verify

npm run build:dataset -- --merge fails loudly if any field is missing or malformed.

Then use the app

Ask Claude to use the wcag-explainer skill — it scaffolds the app and serves it locally.
