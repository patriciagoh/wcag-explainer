How the dataset is built

The app ships a single file — wcag-criteria.json, all 87 success criteria, each in plain English with pass/fail code and grounded common mistakes. It's produced by a three-phase pipeline: fetch the facts, enrich each criterion with Claude, then merge and validate. This page is mostly about the interesting middle phase.

01 Why a pipeline at all

Official WCAG text is precise but impenetrable to a working engineer. The pipeline separates the parts that are data (fetch them, deterministically) from the part that is judgement (let an LLM translate) — and keeps the judgement honest, cached, and validated.

  • ◆ Turn legalese into action. "Non-text content has a text alternative…" becomes "every image needs alt text; decorative ones get alt=""." That rewrite is exactly what an LLM is good at.
  • ◆ Teach with contrast. Each criterion gets a passing and a failing code example. Seeing both side by side teaches faster than prose.
  • ◆ Ground every claim. "Common mistakes" are tied to the criterion's real axe-core rules from the fetch phase, and any real example reuses an actual W3C technique snippet. Facts stay sourced.
  • ◆ Be cacheable & resumable. Each criterion is hashed against its inputs. Re-runs skip anything unchanged, so a refresh only re-enriches what actually drifted.

02 The three phases

Everything lives in scripts/ and is a skill-author task — end users of the app never run it. Fetch and merge are deterministic automation; enrich is the LLM step.

npm run build:dataset -- --fetch

1 · Fetch the facts

Scrapes the W3C Quickref, the linked technique pages, and axe-core rule metadata into scripts/raw/criteria-raw.json. Deterministic; committed.

npm run enrich

2 · Enrich with Claude

For each raw criterion, a scripted call to the Claude API generates the teachable fields into scripts/cache/{id}.json. Incremental — unchanged criteria are skipped.

npm run build:dataset -- --merge

3 · Merge & validate

Joins raw facts + enriched cache, validates every record against a Zod schema, and writes the shipped template/src/data/wcag-criteria.json.

The enrich phase used to be manual — a human driving a Claude Code session by hand. It's now the scripted scripts/enrich.ts (Anthropic SDK, model claude-opus-4-8), behind a mockable client interface so its logic is unit-tested without ever calling the API.

03 Key decisions in the enrich phase

Claude is the "LLM rewrite" step. A few choices keep that honest, reproducible, and reviewable.

Claude is the generator, via a scripted API call

enrichCriterion() fills each prompt template and calls the Claude API through a small EnrichmentClient interface — the same seam a mock implements in the tests.

Why: scripting the call (vs a hand-driven session) makes the whole pipeline reproducible and lets CI exercise the logic against a mock with no API key.

A deterministic inputHash per criterion

SHA-256 over the normalized {officialText, scrapedExamples, axeRules} with recursively sorted keys.

Why: a stable hash lets a run compare against the committed cache and skip criteria whose facts haven't changed — true caching and resume. --reconcile recomputes hashes without calling the API.

Incremental by default, principle at a time

A bare npm run enrich only touches criteria whose inputs changed; --only=2 scopes to one principle, --force re-enriches everything.

Why: a refresh after upstream drift re-bills only what actually changed, and large runs stay easy to checkpoint and commit between principles.

Real example when we have one, synthesize otherwise

Where a criterion has a scraped W3C example, the pass reuses that real snippet; otherwise the pair is synthesized and marked source:"synthesized".

Why: prefer sourced truth, but never ship a criterion without a teaching pair — and label synthesized examples honestly.

Pick the most natural language per example

JSX for component/keyboard patterns, CSS for contrast/motion/target-size, HTML for <meta>/<title>.

Why: forcing everything into JSX would misrepresent CSS-level criteria like contrast or reflow.

Validate against the real schema before writing

Each entry is parsed through the shared Zod cacheEntrySchema before it's written, and the merge re-validates every record on read.

Why: evidence before assertions — a cache file only counts as done if the production validator accepts it.

04 What enrichment produces

For every one of the 87 criteria, a single cache file with six keys — one grounding hash plus five generated fields, each from its own prompt template.

grounding key inputHash

SHA-256 of the criterion's facts. Drives skip/resume on re-runs.

prompt → plain-english.md plainEnglish

2–4 plain sentences explaining what the criterion requires and why.

prompt → why-it-matters.md whyItMatters

One concrete sentence on who is affected when it's violated.

prompt → quick-check.md quickCheck

A one-line test an engineer can run now (DevTools / axe).

prompt → common-mistakes.md commonMistakes[]

3–5 concrete failures, grounded in the criterion's axe rules.

prompt → html-to-jsx.md codeExamples[]

One pass + one fail snippet, each labelled and sourced.

The deliverable: 87 files at scripts/cache/{id}.json, committed alongside the raw facts, then merged into template/src/data/wcag-criteria.json — 87 criteria, fully Zod-validated, every one with a pass and a fail example.

05 How enrichment runs, per criterion

enrich.ts walks the raw criteria; for each stale one it calls enrichCriterion(), which runs the same loop.

  1. Hash & check the cache. inputHash
    Compute the criterion's inputHash. If a cache file exists and the hash matches, skip — already fresh (unless --force).
  2. Generate the three short fields. plain-english · why-it-matters · quick-check
    Fill each prompt template with the criterion's id, title, level, and official text; the three independent calls run concurrently.
  3. Ground the common mistakes. common-mistakes.md
    Feed the criterion's axe-rule descriptions into the template so the 3–5 mistakes reference real, flagged patterns — then parse the JSON array (fences tolerated, validated as strings).
  4. Build the pass/fail pair. html-to-jsx.md
    Pick the best scraped pass & fail examples by label; convert HTML → idiomatic JSX. Where none exist, synthesize a realistic pair and mark it "synthesized".
  5. Validate & write. cacheEntrySchema → cache/{id}.json
    Parse the assembled entry through the Zod schema, then write it. A final --merge re-validates all 87 and ships the dataset.

06 Architecture at a glance

Raw facts and prompt templates flow into a per-criterion loop driven by scripts/enrich.ts. A hash gate skips unchanged work; Claude generates the fields; the result is one validated cache file per criterion, ready for the merge.

  "whyItMatters": "Screen reader users… get nothing useful when an image has no alt text…",
  "quickCheck": "In Chrome DevTools, inspect each <img>… or run the axe DevTools extension.",

08 Run it yourself

A skill-author task — only needed when rebuilding or refreshing the dataset. End users of the app never run it. Enrichment calls the Claude API, so it needs an ANTHROPIC_API_KEY.

scripts/ · terminal
# 1 · fetch the facts (deterministic)
npm run build:dataset -- --fetch

# 2 · enrich with Claude — incremental; skips unchanged criteria
ANTHROPIC_API_KEY=sk-… npm run enrich
#    one principle at a time:  npm run enrich -- --only=1
#    re-enrich everything:     npm run enrich -- --force

# 3 · merge & validate → ships wcag-criteria.json
npm run build:dataset -- --merge
Wrote template/src/data/wcag-criteria.json with 87 criteria

Incremental: the inputHash gate means a re-run skips any criterion whose facts are unchanged — so a refresh after upstream drift only re-enriches what actually changed. npm run enrich -- --reconcile recomputes the hashes without calling the API.

Scope a run

--only=1 … --only=4 enriches one principle; --force ignores the cache.

Verify

npm run build:dataset -- --merge fails loudly if any field is missing or malformed.

No key? Fall back

Hand-write the cache entries and run --reconcile — see scripts/enrich-with-claude.md.
